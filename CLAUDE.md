**Критичные:**

- **PL-15. CORS-preflight сжигает лимит поиска и ломается сам.** `server.js`: лимитеры (`app.use('/api/price-lookup', limiter)` и соседние, строки 587–591) смонтированы ДО CORS-обработчика (строка 594). Виджет магазина зовёт сервис с другого домена (`poigraem.shop` → `api.poigraem.shop`, см. `js/client.js:1703`), а POST с JSON всегда шлёт предварительный OPTIONS-запрос. Итог: каждый поиск из магазина тратит 2 из 10 токенов лимита (OPTIONS + сам POST) — реальный бюджет клиента не 10, а 5 поисков в час, причём общий с заявкой и подпиской (PL-3). Хуже: когда лимит кончился, 429 получает сам OPTIONS-запрос — браузер без CORS-заголовков в ответе видит сетевую ошибку, клиенту показывается «Проверьте соединение» вместо честного «попробуйте через час». У `coversLimiter` защита уже есть (`skip: (req) => req.method === 'OPTIONS'`, строка 585) — у `limiter` и `suggestLimiter` нет. Фикс: переставить блок CORS-middleware выше `app.use('/api/...', limiter)` — тогда OPTIONS отвечается 204 до счётчика лимита.

## Что это

Standalone-сервис «Не нашли игру?» для интернет-магазина «Поиграем» (продаёт цифровые игры PS5/PS4). Клиент вводит название отсутствующей в каталоге игры — сервис автоматически парсит цены через PS Store (Турция и Индия), конвертирует в рубли с наценкой магазина и показывает «ориентировочно от X ₽».

## Архитектурные решения (нерушимые)

### Текущий статус (актуально)

Парсеры и matcher вынесены в `parsers.js`. Сервис **полностью рабочий**:
- `parseSearchPage(html, region)` — парсит `__NEXT_DATA__` со страницы поиска PS Store
- `parsePriceText(text, currency)` — понимает TL/₺ (Турция) и ₹/Rs (Индия)
- `matchGame(query, candidates, opts)` — fuzzy match с биграммами, TR-приоритет, isEditionOfSameGame
- `normalizeQuery(s)` — lowercase, NFD, римские → арабские (кроме x)
- `lookupPrice(gameName)` — параллельный поиск TR+IN, конвертация в RUB, кэш в Supabase

### Скрейпинг чужого сайта — главная боль

PS Store не имеет публичного API. Используем HTML-скрейпинг.

**Это значит:**
- Sony может поменять вёрстку → парсер сломается. Раз в 2-6 месяцев это случается.
- Cloudflare может заблокировать IP → нужен User-Agent обычного браузера + рандомные задержки
- Юридически серая зона → НЕ говори клиенту «данные из PS Store», формулировка «ориентировочно от»

**Делай:**
- Все запросы к Sony — через `fetchHtml()` с реалистичным User-Agent и `Accept-Language`
- Между запросами рандомная задержка 200-800ms (если параллельно — Promise.all с задержками в каждом)
- Кэш в Supabase на 24 часа ОБЯЗАТЕЛЕН — без него быстро забанят
- При 403/5xx/timeout от Sony — деградация к форме-заявке «рассчитаем вручную»

**Не делай:**
- Не дёргай Sony без кэширования
- Не используй Selenium/Puppeteer без крайней необходимости — медленно и тяжело для VPS
- Не указывай клиенту регион из которого цена (TR/IN) — защита механики ценообразования

### Структурированный поиск

PS Store скорее всего рендерит страницу поиска через Next.js. Это значит данные могут быть:

**Вариант A:** в HTML карточках результатов (`<article>`, `<div data-testid="...">`)
**Вариант B:** в `<script id="__NEXT_DATA__">{...JSON...}</script>` в конце страницы

Сначала пробуем найти `__NEXT_DATA__` — там обычно структурированный JSON. Если нет — парсим карточки. Это надо проверить на реальной странице.

### Двухрегиональная логика

Параллельно дёргаем PSN IN и PSN TR. **Это всегда `Promise.all`**, никогда последовательно (×2 latency без выгоды). Каждый регион в отдельной функции с обработкой ошибок — если один регион упал, второй должен сработать.

### Расчёт цены в RUB

Курс валют и `markupTiers` берутся **из общей Supabase магазина** (читаем настройки). НЕ хардкодь курсы. НЕ дублируй логику ценообразования — используй ту же что у магазина.

Формула (аддитивная наценка из тиров):
```javascript
base = priceLocal * rate          // для TRY
base = ceil(priceLocal/1000) * inrGift  // для INR
tier = markupTiers.find(t => base >= t.from && (t.to == null || base <= t.to))
priceRub = ceil((base + tier.markup) / rounding) * rounding
```

`markupTiers` — массив диапазонов `{from, to, markup}` из настроек магазина. `inrGift` — стоимость 1000 INR в рублях.

### Выбор региона (TR-приоритет) — БИЗНЕС-КРИТИЧНО

Обе цены (TR и IN) конвертируются в рубли, дальше **не** берём просто минимум. Турция — приоритетный регион (обычно дешевле и стабильнее по механике гифт-карт), Индия побеждает, **только если её рублёвая цена дешевле турецкой минимум на 12%** (`server.js`, `TR_PRIORITY_FACTOR = 1 - 0.12`): `inRub <= trRub * TR_PRIORITY_FACTOR`.

- **Процент считается ОТ турецкой (большей) цены** — как и все проценты в проекте (скидки PS Store, `WATCH_PRICE_DROP_THRESHOLD`). Не путать со знаменателем «Турция дороже Индии на X%» — это другое число (13.72% от TR = 15.91% от IN на одном и том же разрыве). Меняешь порог — меняй в терминах «дешевле от TR», иначе правило молча поедет.
- Порог живёт в **одной** константе `TR_PRIORITY_FACTOR` и применяется в двух местах: выбор winner (`server.js:~468`) и дедупликация списка изданий (`server.js:~495`). Менять только через константу — иначе winner и список изданий разъедутся.
- Калибровка (сессия 31): порог опущен 20% → 12% под класс «новых игр» вида **3449 TL / 4999 ₹** (разрыв ~13.7%). Маркерный кейс — Marvel's Wolverine: TR 7360 ₽ vs IN 6350 ₽ → при 12% выигрывает Индия (6350 ₽) с запасом ~127 ₽ на движение курса. При 16–20% выигрывала Турция. Порог глобальный: вместе с этим классом на Индию уходят ВСЕ игры в зоне 12–20%.

### Standalone-изоляция

- НЕ модифицируй основной server.js магазина

## Таблицы Supabase

Используем общую Supabase магазина. Price-lookup владеет двумя таблицами и пишет в одну общую:

- **`price_lookup_cache`** — кэш найденных цен. Поиск по `query_normalized`, TTL 24 часа. Структура: `query_normalized, region, concept_id, price_local, currency, price_rub, edition_label, found, fetched_at`
- **`price_lookup_requests`** — заявки «не нашли, рассчитайте вручную». Структура: `game_query, client_contact, contact_type, status, ip_hash, source, created_at, processed_at`
- **`event_tracking`** (общая для магазина+price-lookup+chatbot) — события переходов между сервисами

Полная структура — в `BRIEFING.md` раздел «База данных».

## Контракт API

### POST /api/price-lookup
Запрос: `{ gameName, editionHint?, source? }` где `source` это `direct` или `chatbot`
Ответ (успех): `{ status: "found", priceRUB, neutralLabel, editionLabel?, promo? }`
Ответ (серая зона): `{ status: "ambiguous", candidates: [...] }`
Ответ (не нашли): `{ status: "not_found", suggestion: "manual_request" }`
Ответ (Sony down): `{ status: "store_unavailable", managerLink }`
Ответ (rate limit): `{ error: "rate_limited", message, retry_after }`

### POST /api/price-request
Принимает заявку: `{ gameName, contact, contactType }`. Пишет в `price_lookup_requests`. Опционально шлёт уведомление админу (на старте можно без, потом TG-бот).

### GET /test/product/:region/:conceptId
Отладочный — парсит одну страницу товара. Использовать ТОЛЬКО для разработки парсера. В проде отключить или закрыть авторизацией.

### GET /health
Проверка живости. Возвращает `{ status: "ok", service: "price-lookup", version }`.

## Правила работы с кодом

### Стиль
- Современный JavaScript, ES Modules, async/await
- Без TypeScript (проект небольшой)
- Без лишних абстракций
- Комментарии на русском там где есть смысл объяснить «почему», а не «что»

### Парсинг — defensive
- HTML от Sony может быть каким угодно — всегда try/catch
- Селекторы могут не найти элемент — проверяй на null перед использованием
- Цена может быть «Not available», «Coming soon», «Free» — обрабатывай явно
- Парсеры цены (parseTRPrice/parseINPrice) возвращают `null` если не нашли — не падай
- Логируй проблемы парсинга в console (потом смотрим что Sony поменяла)

### Fuzzy matching
- Нормализация запроса перед поиском: lowercase, trim, убрать пунктуацию, римские → арабские
- Пороги схожести: ≥0.85 автомат, 0.5-0.85 серая зона, <0.5 не нашли
- Для кириллицы порог понижаем (PSN сам умеет в кросс-язычный поиск, доверяем больше ему)

### Защита от блокировок Sony (КРИТИЧНО)
- **User-Agent** — реалистичный Chrome on Windows (уже в коде)
- **Accept-Language** — под регион (en для en-tr, en для en-in)
- **Кэш 24 часа** в Supabase — критично, без него быстро забанят
- **Rate limiting** на эндпоинте — пользователь не должен дёрнуть Sony 100 раз/час через нас
- **Рандомная задержка** 200-800ms между запросами если их много

### Защита механики ценообразования (бизнес-критично)
- НЕ показывай клиенту регион (TR/IN) откуда цена
- НЕ показывай локальную цену (TRY/INR) — только в RUB
- НЕ объясняй как считается наценка
- Формулировка ВСЕГДА «ориентировочно от X ₽», не «точная цена», не «лучшая цена»

## Чего НЕ делать

- **Не дёргай Sony без кэширования** — забанят
- **Не указывай клиенту регион/локальную цену** — защита механики
- **Не используй Selenium/Puppeteer без крайней необходимости** — медленно
- **Не показывай DLC/аддоны в результатах** — фильтр по `productType: GAME`
- **Не показывай игры с маркером "Not available for purchase" / "Included" (PS Plus)** — это не покупка
- **Не давай скидки и не обещай ничего** — компетенция менеджера
- **Не храни сырой IP** — только sha256-hash
- **Не используй наценку отличную от магазина** — единая логика ценообразования через `markupTiers` из настроек

### 2026-07-17 (сессия 31 — фиксы PL-32 (апгрейд-DLC в цене) + PL-33 (порог региона 20%→12%), оба задеплоены и закрыты)

**Контекст:** Денис сообщил, что новая функция магазина «дозаполнить цены» (прайсит закинутые без цены игры) выдала для Marvel's Wolverine 1010 ₽ — подставилась цена не той игры. Claude воспроизвёл на живых страницах PS Store, что виноват парсер price-lookup, а не магазин, и по ходу разбора нашёл второй, отдельный баг ценообразования.

**Что сделано:**
- **PL-32 — апгрейд-DLC выигрывал выбор минимальной цены (`parsers.js`, коммит `1b06454`, задеплоен+подтверждён).** Живой прогон TR+IN: по запросу «Marvel's Wolverine» Sony отдаёт саму игру (FULL_GAME, 3449 TL), Deluxe (PREMIUM_EDITION, 3849 TL) и **«…Digital Deluxe Edition Upgrade» (GAME_BUNDLE, 400 TL)**. Апгрейд по classification неотличим от Ultimate-издания → проходил фильтр «только игры», а `isEditionOfSameGame` роднил его с игрой → все три попадали в один ценовой пул → выбор минимума забирал апгрейд за 400 TL (≈1010 ₽). Две причины, обе нужны: (1) `GAME_BUNDLE` в белом списке; (2) `normalizeQuery("Marvel's Wolverine")` из-за апострофа давал «marvel s wolverine» — 3 слова, из-за чего срабатывало правило «≥3 слов → остаток подзаголовок». Фикс двумя заходами: **А** — `UPGRADE_RE = /\bupgrade\b/` отсекает кандидатов со словом upgrade в имени (если сам клиент апгрейд не искал), стоит в первичном фильтре до развилки trustTop; **Б** — апострофы (`'’‘\`´`) удаляются в `normalizeQuery` без пробела («marvels wolverine», 2 слова), заодно чинит биграммное сходство тайтлам Assassin's/Tom Clancy's. Новый юнит-тест 2d (6/6), обновлён ассерт normalizeQuery (`assassins creed`). Регрессия на живых страницах Wolverine/Spider-Man 2/AC Mirage/Hogwarts прогнана через версию из HEAD и исправленную — изменился ровно Wolverine.
- **PL-33 — порог TR-приоритета 20% → 12% (`server.js`, коммит `394bd52`, задеплоен+подтверждён).** Обе цены (TR/IN) конвертируются в рубли, дальше берётся не минимум: Турция приоритетна, Индия побеждает только если дешевле на ≥X% (`inRub <= trRub * factor`). Класс новых игр вида 3449 TL / 4999 ₹ имеет разрыв ~13.7% — при пороге 20% Индия проигрывала, клиент видел завышенную турецкую цену. Порог опущен до 12% (Wolverine: TR 7360 → IN 6350, запас ~127 ₽ над классом). Порог вынесен в единую константу `TR_PRIORITY_FACTOR = 1 - 0.12`, применяется и в выборе winner, и в дедупликации изданий (раньше два литерала `0.8`). Правило задокументировано в CLAUDE.md (раздел «Выбор региона» в «Архитектурных решениях») — раньше его не было нигде.
- **Незакоммиченный `/session-end` сессии 30 закоммичен** (коммит `898e45f`): VPS-агент при деплое PL-32 читал CLAUDE.md с GitHub, где доки отставали на 13 июля, и принял деплой PL-31 за недокументированный — оказалось, session-end сессии 30 просто висел незакоммиченным. Заодно в шаблон деплоя внесена поправка: `node --check файл.new` НЕ работает при `"type": "module"` (Node не выводит модульность для расширения `.new`), проверять синтаксис на копии `.mjs` (нашёл VPS-агент).
- **Деплой обоих фиксов (VPS-агент, доказательные отчёты):** PL-32 — 1010 ₽ (апгрейд) → 7360 ₽ (сама игра), тесты 6/6, регрессия чистая. PL-33 — 7360 ₽ (TR) → 6350 ₽ (IN), `[getSettings] rate=1.76 inrGift=1050` из Supabase совпал с расчётным, три контрольные игры остались в Турции (там дешевле). Дрейфа VPS↔репо нет — деплой был обновлением из origin/main.

**Что решили:**
- Оба фикса (А+Б для PL-32) реализованы одним заходом по рекомендации Claude и выбору Дениса; вариант «ужесточить правило ≥3 слов» отклонён как рискованный (задевает легитимные «The Witcher 3 Wild Hunt»).
- Порог PL-33 = 12% в **текущей** формуле (процент от турецкой, большей цены — как все проценты в проекте). Разобрано, что формулы «Индия дешевле TR на X%» и «TR дороже IN на X%» эквивалентны, но дают разные числа на одном разрыве (13.72% vs 15.91%); чтобы правило не поехало молча, знаменатель зафиксирован в коде и доке. 12% выбран из-за запаса ~127 ₽ над маркерным классом (13% давал всего 53 ₽ — при движении курса игры прыгали бы между регионами).
- PL-32 и PL-33 закрыты полностью (задеплоены + живой прогон) → перенесены в `AUDITS-ARCHIVE.md`, в отличие от прошлых «код готов, но не проверено».
