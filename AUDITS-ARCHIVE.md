## Закрытые находки внутри аудита PL-23…PL-30 (сессия 29)

### ✅ PL-23 — закрыт, сессия 29 (код) + сессия 30 (живой прогон), коммиты `0b2ef15` + `eec6cc4`

**Было:** `parsers.js`, режим `trustTop` (кириллический запрос вне `CYRILLIC_TRANSLATE`, не переведённый IGDB): `alternatives = eligible.slice(1, 4)` — соседние результаты поиска Sony попадали в список «изданий» без проверки, что это действительно та же игра. `lookupPrice` сваливал `best` + `alternatives` обоих регионов в общий ценовой пул и мог выбрать минимум с чужим названием — клиенту показывалась посторонняя игра как «издание».

**Фикс:** `isEditionOfSameGame(queryNorm, nameNorm, strict)` получил параметр `strict` (отключает правило «запрос ≥3 слов → остаток это подзаголовок», которое при сравнении двух ПОЛНЫХ имён товаров ложно роднило сиквелы, напр. «god of war ragnarok» сходило бы за издание «god of war»). В trustTop-ветке `alternatives` теперь фильтруются двусторонней strict-проверкой относительно `best`. Юнит-тест 2b в `test-parsers.js` (синтетические кандидаты, не требует `recon/`).

**Подтверждение закрытия:**
- Юнит-тест 2b — 4/4 локально и на живом задеплоенном файле VPS (сессия 29).
- Живой trustTop-прогон (сессия 30, попутно с проверкой PL-31): запрос «резидент ивел 4» реально прошёл через ветку `trustTop` на проде и отработал корректно (см. PL-31 ниже) — единственное, чего не хватало для закрытия PL-23 («живой запрос через trustTop-ветку не прогонялся»), теперь выполнено.

⚠️ Сам фикс PL-23 (фильтр `alternatives` относительно `best`) не защищал сам `best` — эта дыра осталась и её отдельно закрыл PL-31 (см. ниже) в сессии 30.

### ✅ PL-31 — закрыт, сессия 30, коммит `74854ae`

Найден не в ходе формального аудита, а во время приёмки пакетов сессии 28: Денис попросил Claude выполнить визуальную/API-приёмку живого сайта (iPhone для клик-теста не было), и в процессе живого прогона кириллических запросов обнаружился баг.

**Было:** в режиме `trustTop` (`parsers.js`) `best = eligible[0]` брался **безусловно**, без всякой проверки релевантности запросу — `similarity()` (биграммное Dice-сходство) там бесполезен, т.к. кириллический запрос и латинское имя кандидата никогда не пересекаются по биграммам (это и была причина, по которой Dice сознательно пропускался в trustTop). Живой пример: запрос «резидент ивел 4» (Resident Evil 4, транслитерация нестандартная, не в словаре `CYRILLIC_TRANSLATE`, IGDB не перевёл) — сырой кириллический поиск PS Store вернул слабые/нерелевантные совпадения, и первое из них («4PGP», посторонний товар) уверенно отдавалось клиенту как «found» с ценой 1990 ₽. Фикс PL-23 (см. выше) защищал только `alternatives` относительно `best` — сам `best` не проверялся никогда, поэтому баг мог всплыть на любом кириллическом запросе с нестандартной транслитерацией или редкой игрой, не только на этом конкретном названии.

**Решение (вариант выбран Денисом из трёх предложенных):** новая `translitCyrillic()` — грубая фонетическая карта кириллица→латиница (а→a, ж→zh, щ→sch и т.д.). В trustTop-ветке кандидаты теперь **переранжируются** по `similarity(translit(query), normalizeQuery(name))`, а не берётся первый по позиции Sony:
- сходство ≥ `TRUST_CONF` (0.45) → `found` (уверенный матч);
- `TRUST_GREY` (0.28) ≤ сходство < `TRUST_CONF` → `ambiguous` (клиент выбирает сам через «вы имели в виду?», а не сервис угадывает);
- сходство < `TRUST_GREY` → `not_found` (честная заявка менеджеру лучше уверенно неправильной цены).

Фильтр изданий PL-23 вынесен в переиспользуемый хелпер `trustTopEditions()`.

**Подтверждение закрытия (сессия 30, VPS-агент):**
- Юнит-тест 2c (синтетические кандидаты, не требует `recon/`) — 5/5: best = «Resident Evil 4» (не «4PGP»), «4PGP» не в alternatives, Deluxe-издание корректно осталось в alternatives, чисто мусорный запрос → `not_found`, а не `found` наугад.
- Регрессия проверена: тест 2b (PL-23) по-прежнему 4/4.
- **Живой прогон до/после рестарта на VPS:** ДО рестарта (старый код) реальный запрос «резидент ивел 4» дважды подряд возвращал «4PGP» с ценой — баг воспроизведён на проде. ПОСЛЕ рестарта тот же сырой кандидат «4PGP» по-прежнему приходит от Sony, но rerank его больше не пропускает → честный `not_found` вместо чужой цены.
- Пороги `0.45`/`0.28` эвристические (транслитерация фонетическая, неточная — например «резидент» даёт «rezident» вместо «resident»), откалиброваны на здравом смысле и синтетических тестах, не на реальном трафике — стоит последить за первое время.

### ✅ PL-19 — закрыт, сессия 28, коммиты `0eafa89` (код) + Supabase GRANT (Денис)

**Было:** `POST /api/price-watch/init` писал `priceRub` из тела запроса напрямую в `price_at_subscribe_rub` без валидации диапазона/типа — можно было подписаться с искусственно завышенной ценой и получить гарантированное «подешевело» при первой проверке. Плюс: pending-записи (`active:false`, если клиент не нажал Start в боте) не удалялись автоматически — таблица `price_watchlist` копила мусор.

**Фикс:**
- `server.js` — валидация `priceRub` (`Number.isFinite`, `0 < x ≤ 100000`) перед записью, иначе 400.
- Новая `cleanupStaleWatches()` — удаляет строки `active:false AND tg_chat_id IS NULL AND notified_at IS NULL AND created_at < now()-7d`; вызывается в конце каждого прогона `checkWatchlist()`. История уже сработавших уведомлений (`notified_at` не пуст) не трогается.

**Скрытая зависимость, найденная при деплое:** `cleanupStaleWatches()` требовала `GRANT DELETE ON price_watchlist TO service_role` — таблица создана через SQL Editor без дефолтных грантов (тот же класс проблемы, что в сессии 21 для SELECT/INSERT/UPDATE, тогда DELETE просто забыли). Денис выполнил грант вручную в Supabase SQL Editor.

**Подтверждение закрытия (живой прогон, сессия 28):**
- Валидация: `priceRub: 999999999` → `400`, запись в БД не создана (проверено SELECT'ом — единственная тестовая строка «Test Game» в таблице имела цену 2500, не 999999999).
- Грант: после `pm2 restart price-lookup` в `error.log` нет свежих `permission denied` (единственная такая строка в файле — с меткой времени ДО гранта).
- Очистка реально работает: `out.log` после рестарта содержит `[watchlist] cleaned up 1 stale pending subscriptions` — функция нашла и удалила настоящую брошенную запись.
